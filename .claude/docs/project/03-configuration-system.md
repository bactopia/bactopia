# Configuration System

## Overview
The Bactopia pipeline uses a hierarchical configuration system that allows settings to be overridden at multiple levels, providing flexibility while maintaining consistency across executions.

## Configuration Hierarchy

Configuration values are inherited in the following order (highest to lowest priority):
1. **Command line arguments** - Direct overrides at runtime
2. **Workflow-specific config** - `{workflow}/nextflow.config`
3. **Module config** - `{tool}/module.config`
4. **Global params** - `conf/params.config`
5. **Default values** - Hardcoded defaults (lowest)

## Main Configuration Files

### Root Configuration
- **`nextflow.config`** - Global pipeline configuration
  - Process definitions and resource allocation
  - Profile configurations (conda, docker, singularity, etc. — see [Profile Configuration](#profile-configuration) below for the full list)
  - Timeline and report settings
  - Manifest for Docker/Singularity images

### Base Configuration
- **`conf/base.config`** - Base settings for all executions
  - Default process labels and container configurations
  - Basic parameter definitions
  - Default resource limits

### Profile Configuration
- **`conf/profiles.config`** - Execution environment profiles
    - `standard` - Default; local execution with standard resource allocation
    - `conda` / `mamba` - Conda environment management
    - `docker` / `podman` - Docker-compatible container execution
    - `singularity` / `apptainer` / `charliecloud` / `shifter` - HPC container runtimes
    - `wave` - Seqera Wave container builds
    - `arm` - ARM-architecture container variants
    - `gitpod` - Gitpod cloud IDE
    - `test` - Test-suite resource limits

### Parameter Configuration
- **`conf/params.config`** - Global parameter defaults
- **`catalog.json`** - Component catalog (modules, subworkflows, workflows) — lives at the repo root
- **`conf/params/{workflow}.config`** - Workflow-specific parameters
- **`conf/schema/`** - Auto-generated JSON schemas for parameter validation

## Parameter System

### Parameter Sources
Parameters flow through multiple locations:
1. **Global defaults** - `conf/params.config`
2. **Module defaults** - `{tool}/module.config`
3. **Workflow declarations** - `{workflow}/main.nf` (typed `params { ... }` block at the top of named workflows — e.g., `workflows/cleanyerreads/main.nf`, `workflows/staphopia/main.nf`, `workflows/teton/main.nf`)
4. **Workflow validation** - `{workflow}/nextflow_schema.json`
5. **Runtime overrides** - Command line arguments

### Schema Validation
- **Main schema** - `nextflow_schema.json` (deprecated, workflow-specific preferred)
- **Workflow schemas** - `{workflow}/nextflow_schema.json`
- **Module schemas** - `{tool}/schema.json`
- Generated by merging module schemas using `bactopia-merge-schemas`

### Parameter Types

**Primitive params** (used in `params { ... }` blocks across `conf/params.config`, `module.config`, and workflow `main.nf`):
- `String` - Text values
- `Integer` - Whole numbers
- `Boolean` - True/false flags
- `Path` - File paths (required)
- `Path?` - Optional file paths (null when not provided)

**Subworkflow `take:` types** (channel-typed inputs — see [Technical Specifications](../standards/03-technical-specs.md)):
- `Channel<Record>` - Per-sample record streams from upstream modules/subworkflows
- `Value<Path>` - Required shared inputs (databases, reference genomes)
- `Value<Path?>` - Optional shared inputs

## Module Configuration

Each module includes configuration files:

### `module.config`
Combined configuration file containing both parameters and process settings:
```groovy
params {
    // tool_name
    tool_option = "default_value"
    tool_threshold = 80
    tool_database = "default_db"
}

process {
    withName: 'TOOL_NAME' {
        ext.wf = params.wf
        ext.scope = "sample"
        ext.subdir = ""
        ext.logs_subdir = ""
        ext.process_name = "tool-name"

        // Tool arguments
        ext.args = [
            "--option1 ${params.tool_option}",
            "--threshold ${params.tool_threshold}"
        ].join(' ').replaceAll("\\s{2,}", " ").trim()

        // Environment information
        ext.toolName = "bioconda::tool-name=1.0.0".replace("=", "-").replace(":", "-").replace(" ", "-")
        ext.docker = "biocontainers/tool-name:1.0.0--hash_0"
        ext.image = "https://depot.galaxyproject.org/singularity/tool-name:1.0.0--hash_0"
        ext.condaDir = "${params.condadir}"
    }
}
```

### `schema.json`
JSON schema for parameter validation:
```json
{
    "$schema": "https://json-schema.org/draft/2020-12/schema",
    "title": "Module Name",
    "type": "object",
    "properties": {
        "tool_option": {
            "type": "string",
            "default": "default_value",
            "description": "Description of the option"
        }
    }
}
```

## Workflow Configuration

Each workflow includes:
- **`nextflow.config`** - Workflow-specific settings
- **`nextflow_schema.json`** - Parameter validation schema

### Workflow Config Pattern
```groovy
// Workflow manifest
manifest {
    name = 'bactopia-tool-name'
    author = 'Your Name'
    homePage = 'https://github.com/bactopia/bactopia'
    description = 'Brief description of the tool'
}

// Workflow parameters
params {
    // Include workflow-specific parameters
    rundir = "${params.outdir}/bactopia-tools/${params.workflow.name}"
}

// Include required configs
//
// Nesting depth depends on workflow location:
//   - Named workflows (`workflows/{name}/`)              — 2 levels back: `../../conf/...`, `../../modules/...`
//   - Bactopia-tools (`workflows/bactopia-tools/{name}/`) — 3 levels back: `../../../conf/...`, `../../../modules/...`
//
// Example for a bactopia-tool (3 levels back):
includeConfig '../../../conf/base.config'
includeConfig '../../../conf/params.config'
includeConfig '../../../conf/profiles.config'
includeConfig '../../../conf/params/bactopia-tools.config'

// Include module configs (same 3-level pattern)
includeConfig '../../../modules/{tool}/module.config'
```

### The `params.workflow` Object

Every workflow's `nextflow.config` populates a `params.workflow` block that carries workflow-identity metadata consumed by utility subworkflows (e.g., for `rundir` construction, report filenames, and tool-manifest building):

```groovy
params {
    workflow {
        name        = 'abricate'
        logo_name   = 'abricate'
        description = 'Mass screening of contigs for antimicrobial and virulence genes'
        ext         = ['tsv']  // bactopia-tools only: output file extensions to publish
    }
}
```

Keys:
- `name` - Workflow identifier (matches the workflow's directory name)
- `logo_name` - Name used when rendering the startup logo
- `description` - One-line description (shown in help text and reports)
- `ext` - *(bactopia-tools only)* Array of output file extensions the workflow publishes

Referenced throughout the pipeline as `${params.workflow.name}`, `${params.workflow.description}`, etc.

## Configuration Best Practices

### Parameter Naming
- Use descriptive names (e.g., `kraken2_database` not `db`)
- Follow snake_case convention
- Include tool prefix to avoid conflicts
- Parameters must be in **alphabetical order** within the params block

### Default Values
- Provide sensible defaults
- Optional `Path?` parameters default to null when not provided

### Resource Allocation
Use standard process labels:
- `process_low` - Low resource requirements
- `process_medium` - Medium resource requirements
- `process_high` - High resource requirements
- `process_long` - Long-running processes

### Container Management
- Always specify `ext.docker`, `ext.image`, and `ext.toolName` in module.config
- Container resolution is handled globally in `conf/base.config` (sets `ext.container`)
- Modules use `container "${task.ext.container}"` — never inline ternary logic

## Common Configuration Patterns

### Conditional Parameters
```groovy
params {
    // Conditional parameter with default
    use_optional_tool = false
    optional_tool_db = null
}
```

### Profile-Specific Settings
```groovy
profiles {
    conda {
        conda.createFiles = false
    }
    singularity {
        singularity.autoMounts = true
    }
    docker {
        docker.userEmulation = true
    }
}
```
